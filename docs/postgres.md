# Pedantic notes on PostgreSQL (using this codebase)

Examples below are taken verbatim from the schema, ingest, and workflow queries. The goal is to be precise about behavior, not exhaustive.

---

## 1. Composite natural keys: what indexes you get for free

We chose natural keys, so a child row like `bbox_predictions` has a four-column PK:

```sql
PRIMARY KEY (patient_external_id, embryo_label, timepoint, model_version)
```

Postgres creates one **btree** index for this PK. That index supports any query whose `WHERE` clause is a **leading prefix** of those columns:

| Query predicate | Uses PK index? |
|---|---|
| `WHERE patient_external_id = ? AND embryo_label = ?` | yes |
| `WHERE patient_external_id = ? AND embryo_label = ? AND timepoint = ?` | yes |
| `WHERE patient_external_id = ? AND timepoint = ?` | **no** (skips `embryo_label`) |
| `WHERE timepoint = ?` | no (no leading column fixed) |

It also supports `ORDER BY timepoint` *after* the leading columns are fixed by equality — this is why `workflow_3_morphokinetic_timeline.sql` can sort 923 rows in milliseconds without an extra index. If you ever need to query predictions globally by timepoint or model_version, you'll add a secondary index — there is none today because no workflow asks for it.

## 2. `ON DELETE CASCADE`: DB-level vs ORM-level, and why both

Our composite FK from `bbox_predictions` to `embryos`:

```sql
FOREIGN KEY (patient_external_id, embryo_label)
  REFERENCES embryos(patient_external_id, label)
  ON DELETE CASCADE
```

Without that clause, deleting an embryo with predictions raises `ForeignKeyViolation`. With it, Postgres deletes the children inside the same transaction.

The ORM has its own cascade machinery. In `backend/db/models.py` we use **both**:

```python
images: Mapped[list["Image"]] = relationship(
    back_populates="embryo", cascade="all", passive_deletes=True
)
```

`passive_deletes=True` tells SQLAlchemy *not* to enumerate children and issue per-row `DELETE`s when you delete the parent through the ORM — it trusts the DB cascade. Without `passive_deletes`, you'd get a SELECT-then-N-DELETEs storm for what Postgres could do in one statement. The pairing of DB-level `CASCADE` + ORM `passive_deletes=True` is the canonical idiom.

`cascade="all"` (without `delete-orphan`) propagates `save-update` and `merge` through the relationship so `patient.embryos.append(e)` enrolls `e` in the session.

## 3. ENUM types: order is part of the schema

Postgres `ENUM` types are stored as **4 bytes per value** (an OID). You can confirm:

```
$ psql -d embpred -c "\dT+ morphokinetic_stage"
 Schema |        Name         | Size | Elements
 public | morphokinetic_stage |    4 | t1, tPN, tPNf, t2, ..., tEmpty
```

The order in `CREATE TYPE ... AS ENUM ('t1', 'tPN', ...)` defines both display order and comparison order:

```sql
SELECT 'tB'::morphokinetic_stage > 'tM'::morphokinetic_stage;  -- true
```

This is exactly the biological order, so `ORDER BY stage` is meaningful without joining a lookup table.

Pitfalls:
- **Adding a value mid-list requires `ALTER TYPE ... ADD VALUE 'foo' BEFORE 'bar'`**, which rewrites every dependent column's storage in newer PG versions, and is non-transactional in older versions.
- **Renaming a value** needs `ALTER TYPE ... RENAME VALUE 'old' TO 'new'` (PG 10+).
- **Removing a value** is not supported. You must dump, drop, recreate, reload.

That's why we set `create_type=False` on the SQLAlchemy enum and manage creation/drop explicitly in Alembic — autogenerated migrations don't know how to do these dances safely.

## 4. The `box` type: Postgres normalizes your corners

We store axis-aligned bounding boxes in a single `box` column (16 bytes, two `point`s). Postgres re-orders the two stored points into a canonical form on write — empirically, "upper-right then lower-left":

```
INSERT  ... bbox = (360, 221), (713, 598)
SELECT  ...
         bbox
─────────────────────
 (713,598),(360,221)
```

The integers are identical; the *labels* in the stored representation differ. Our `Box.result_processor` in `backend/db/types.py` recovers ul/lr by min/max:

```python
return BBox(min(x1, x2), min(y1, y2), max(x1, x2), max(y1, y2))
```

If you treat box's stored order as authoritative, you'll silently flip rectangles. Always normalize on read.

We also use `bind_expression` to wrap the bound parameter in `CAST(:p AS box)`. Without that, psycopg sends a plain string and Postgres doesn't coerce it for typed columns automatically.

## 5. `TIMESTAMPTZ`, `TIMESTAMP`, `DATE`: which one and why

Three time-shaped columns in this schema, each picked deliberately:

```
patients.created_at      TIMESTAMPTZ  default now()
embryos.date_seeded      DATE
ploidy_predictions.predicted_at  DATE  default CURRENT_DATE
```

- `TIMESTAMPTZ` (alias `timestamp with time zone`) is 8 bytes; despite the name it does *not* store a timezone. It stores microseconds since the epoch normalized to UTC. The "with time zone" only affects how text input/output is converted relative to the session's `TimeZone` setting. Use it for any event whose absolute moment matters across timezones — like `created_at`.
- `TIMESTAMP` (without TZ) is also 8 bytes but stores the literal wall-clock with no UTC interpretation. Use only when "Tuesday at 9am local" is the truth (rare in backend services).
- `DATE` is 4 bytes and has no time component. `date_seeded` and `predicted_at` are calendar dates — there's no meaningful "time of day" — so `DATE` is the honest type.

A common bug: `created_at TIMESTAMP DEFAULT now()`. `now()` returns `TIMESTAMPTZ`, so Postgres silently strips the offset, and your "5pm" rows from PST and EST sort identically even though they're an hour apart. Use `TIMESTAMPTZ` *or* a `TIMESTAMP DEFAULT (now() AT TIME ZONE 'UTC')` — never the lossy combination.

## 6. Server defaults: `now()` vs `CURRENT_DATE` vs `CURRENT_TIMESTAMP`

| Function | Returns | Notes |
|---|---|---|
| `now()` | `timestamptz` | PG-idiomatic; same as `CURRENT_TIMESTAMP` |
| `CURRENT_TIMESTAMP` | `timestamptz` | SQL-standard alias for `now()` |
| `CURRENT_DATE` | `date` | Today's date in the session timezone |
| `LOCALTIMESTAMP` | `timestamp` (no TZ) | Wall clock |
| `clock_timestamp()` | `timestamptz` | Refreshed every call; the others are frozen for the whole transaction |

`now()` is constant within a transaction — useful for `created_at = updated_at` on insert, surprising if you `SELECT now(); SELECT pg_sleep(1); SELECT now();` and get the same value twice. Use `clock_timestamp()` if you want wall-clock progress.

Server defaults run when the column is omitted from the INSERT *or* when SQLAlchemy is asked to use the default. Passing an explicit `None` will write `NULL` and *bypass* the default — easy to do by accident from Python.

## 7. `USING` vs `ON` in joins: the column-name asymmetry trap

Several times during ingest verification I wrote:

```sql
JOIN images i USING (patient_external_id, embryo_label)
-- ERROR: column "embryo_label" specified in USING clause does not exist in left table
```

`USING` requires that both tables have a column with the **same name**. Our schema has `embryos.label` but `images.embryo_label` — intentional asymmetry: on the embryo table itself `label` is unambiguous; on a child table it would clash with the child's own concepts, so we prefix.

The fix is explicit `ON`:

```sql
JOIN images i
  ON i.patient_external_id = e.patient_external_id
 AND i.embryo_label        = e.label
```

A bit more verbose, but it survives column renames and never lies about which columns it's matching. The workflow query files all use `ON`.

## 8. Lexicographic ordering on TEXT: why `Emb10` sorts before `Emb2`

```sql
SELECT label FROM embryos ORDER BY label;
-- Emb1, Emb10, Emb2, Emb3, ...
```

`TEXT` comparisons are byte-by-byte under the collation (`en_US.UTF-8` by default on macOS Postgres). `'1'` < `'2'` < ... so `Emb10` ≤ `Emb1X` for any `X` starting with `0`.

Three remedies, in increasing complexity:

1. **Sort in the application.** That's what `scripts/ingest.py:_embryo_sort_key()` does — extracts the integer suffix and sorts numerically.
2. **`ORDER BY substring(label from '\d+')::int`** — works at the SQL level but couples the sort to the label format.
3. **Add a numeric column** (e.g. `embryos.position`) and sort by that. Required if you'll ever expose pagination by sort order.

We picked (1) because no UI surface currently sorts embryos by label SQL-side; the demo orders by `live_birth DESC` instead.

## 9. `psql` variable substitution: `:var`, `:'var'`, `:"var"`

Our workflow SQL files start like:

```sql
\set patient_id 'patient1'

WHERE pt.external_id = :'patient_id'
```

Three substitution forms in psql:

| Syntax | Effect | When to use |
|---|---|---|
| `:varname` | Raw substitution, no quoting | Numeric literals: `WHERE timepoint = :timepoint` |
| `:'varname'` | Single-quoted string literal | String values: `WHERE label = :'embryo_label'` |
| `:"varname"` | Double-quoted identifier | Dynamic column/table names: `ORDER BY :"sort_col"` |

If you write `WHERE x = :patient_id` (no quotes) with `\set patient_id 'patient1'`, psql substitutes `patient1` literally, producing `WHERE x = patient1` — and Postgres tries to resolve `patient1` as a column. The quoted form `:'patient_id'` produces `WHERE x = 'patient1'`, which is what you want for a literal string.

Conditional defaults (so `psql -v patient_id=other -f q.sql` overrides):

```sql
\if :{?patient_id}
\else
  \set patient_id 'patient1'
\endif
```

We don't use this in the demo (hard-coded defaults are fine for living docs) but it's the canonical pattern if you parameterize.

## 10. `NULLS LAST` in `ORDER BY`

```sql
ORDER BY lb.score DESC NULLS LAST
```

Postgres' default for `DESC` is `NULLS FIRST` and for `ASC` is `NULLS LAST`. If you `LEFT JOIN` predictions onto embryos and a few rows have no prediction, `ORDER BY score DESC` will float those NULL embryos to the top — which is the opposite of what "best embryos first" should mean. Always be explicit when nulls are possible.

## 11. `REAL` vs `DOUBLE PRECISION` for probabilities

`ploidy_predictions.score` and `live_birth_predictions.score` are `REAL` (4 bytes, ~7 decimal digits of precision). For a `[0, 1]` probability, that's plenty:

- `0.7300000190734863` is the closest `REAL` to `0.73` — round to 2 decimals at display and you'll never see the noise.
- `DOUBLE PRECISION` (8 bytes) would give 15+ digits; for a probability score that's wasted bytes per row across millions of rows.

Our `CHECK (score >= 0 AND score <= 1)` constraint runs on every insert/update. It rejects `NaN`, but **not** for the reason IEEE 754 would suggest: Postgres deliberately departs from IEEE and treats `NaN` as greater than every non-NaN value. So `NaN >= 0` is `true` (NaN wins), but `NaN <= 1` is `false` (NaN is bigger than 1), and the conjunction fails. `IS NOT NULL` won't catch `NaN` either way — NaN is a value, not a null.
